# Writing matchers with your coding agent

The intended loop:

```
scan (fast, wide) → process (AI, slow + expensive) → revalidate → write better matchers
```

Most of the built-in matchers were generated by handing a confirmed
finding to a coding agent and asking it to write a regex matcher for
sites of the same shape. The prompts below work in any agent that can
read your repo and write files (Claude Code, Codex, Cursor, Aider,
Continue, etc.).

## When to write one

- A revalidated true-positive needs a matcher to catch siblings on future scans.
- A cluster of `other-*` slugs in `deepsec metrics` points at a real category deepsec has no name for.
- You want pre-AI noise reduction on a shape that's costly to re-investigate.
- You have an **organization-specific** pattern (internal auth helper, internal SDK call, custom middleware). Those go in a plugin; see [plugins.md](plugins.md).

CWE-shaped or framework-shaped patterns go in
`packages/scanner/src/matchers/`. Codebase-specific or company-internal
patterns go in your plugin.

## Workflow

### 1. Find a finding worth generalizing

After a process + revalidate pass, export the markdowns:

```bash
pnpm deepsec export --project-id my-app --format md-dir --out ./findings \
  --only-true-positive --min-severity HIGH
```

Open one. If you'd want to catch more sites of the same shape on the
next scan, it's matcher-worthy.

### 2. Hand the finding to your agent

Copy this prompt:

> I want to add a matcher to deepsec (this repo) that catches code shaped
> like the finding in `<paste path to the finding markdown>`.
>
> Read these files first to understand the contract:
> - `packages/core/src/plugin.ts` — the `MatcherPlugin` interface
> - `packages/scanner/src/matchers/sql-injection.ts` — a small precise matcher
> - `packages/scanner/src/matchers/auth-bypass.ts` — a small normal-tier matcher
> - `packages/scanner/src/matchers/utils.ts` — the `regexMatcher` helper
> - `samples/webapp/matchers/webapp-debug-flag.ts` — a small worked plugin matcher
>
> Then write a new matcher:
> 1. Decide the slug (kebab-case, names what it flags, e.g. `oauth-state-reuse`).
> 2. Pick a noise tier:
>    - `precise` — pattern only matches the vulnerable shape, minimal FPs.
>      Use when you can describe the bug exactly.
>    - `normal` — broader, the AI does the disambiguation. Default.
>    - `noisy` — very wide net, used to force AI review of a path glob.
> 3. Write the regex(es). Skip test files (`.test.`, `.spec.`, `__tests__`,
>    `_test.go`, etc.). Tighten `filePatterns` so it only globs files where
>    the pattern can occur.
> 4. Save to `packages/scanner/src/matchers/<slug>.ts` (or the plugin's
>    matchers/ directory if it's org-specific).
> 5. Register it in `packages/scanner/src/matchers/index.ts` (import +
>    `registry.register(...)`).
> 6. Run `pnpm deepsec scan --project-id <id> --root <path> --matchers <slug>`
>    and report how many candidates fired. Read 3 of them to spot-check
>    the regex isn't producing obvious FPs.
> 7. Run `pnpm test` and `pnpm lint`. Fix whatever they complain about.
>
> Bias toward `precise` unless the pattern is genuinely ambiguous. Always
> set `filePatterns` tightly — a `noisy`-tier matcher with `**/*.{ts,tsx}`
> can wedge the scanner on a 100k-file repo.
>
> The finding I want to generalize is below. Generalize the *shape* of the
> bug, not the specific identifiers — e.g. if the finding is about
> `getThing(req.query.id)`, the matcher should catch any `get*(req.query.*)`,
> not literal `getThing`.

Paste the finding's markdown after that block. The agent reads the
referenced files (the contract is short — ~60 lines for the interface,
~30 lines for the helper) and writes a matcher.

### 3. Run it, tune it, ship it

```bash
pnpm deepsec scan --project-id my-app --root ../my-app --matchers <new-slug>
```

Watch the candidate count. 0 means too strict (loosen). >100 in a small
repo means too loose (tighten). Typical sweet spots: 1–20 hits per 1k
files for `precise`; 5–100 for `normal`.

If you like it, commit. The next full scan picks it up.

## Noise tiers

| Tier | When | Example |
|---|---|---|
| `precise` | Pattern is unambiguous. | `prisma-raw-sql`: `\$queryRawUnsafe\s*\(` matches only the unsafe API. |
| `normal` | Pattern is broader; AI disambiguates. | `auth-bypass`: flags admin checks and skip-auth strings; AI judges. |
| `noisy` | Every file matching a glob should be reviewed by the AI. | `service-entry-point`: every `**/api/**/route.ts` becomes a candidate. |

Tier also influences ordering. `precise` candidates are processed first
because they have the highest signal per token.

## File globs

Set `filePatterns` tightly. A noisy matcher with `**/*.{ts,tsx}` wedges
the scanner on a 100k-file repo. Prefer:

- Language-specific: `**/*.go`, `**/*.lua`, `**/*.tf`
- Directory-anchored: `**/api/**/*.ts`, `**/services/**/handlers/*.ts`
- Combined: `**/services/**/*.{ts,go}`

## Generic vs plugin

Decision tree:

| Catches… | Where |
|---|---|
| A CWE shape (path traversal, SSRF, prototype pollution) | `packages/scanner/src/matchers/` |
| A framework shape (Next.js, Drizzle, tRPC, Prisma) | `packages/scanner/src/matchers/`, `framework-` or `<lib>-` prefix |
| An org-specific helper, package, or route layout | Your plugin's `matchers/` |
| Mentions a concrete internal service name | Your plugin |

When in doubt, start in core. A reviewer will move it to a plugin on the
PR if it's too org-specific. See [plugins.md](plugins.md) for plugin
authoring.

## Example: how `prisma-raw-sql` got added

1. **Finding.** A scan against an open-source scheduling app surfaced a
   `$queryRaw` call site whose argument was being built by string
   concatenation. The agent revalidated it as a HIGH true-positive.
2. **Generalization.** The vulnerable shape: "any call to a Prisma raw
   SQL escape hatch where the string is built dynamically." Variants
   include `$queryRaw(...)` (vs the safe tagged-template form),
   `$queryRawUnsafe`, `$executeRaw(...)`, `$executeRawUnsafe`, and
   `Prisma.raw()` (which bypasses parameterization wholesale).
3. **Matcher.** `packages/scanner/src/matchers/prisma-raw-sql.ts`.
   Seven regex patterns, `precise` tier, glob
   `**/*.{ts,tsx,js,jsx,mjs,cjs}`. ~30 lines.
4. **Result.** Re-scanning the same app caught the original finding plus
   six siblings the AI hadn't gotten to yet.

End-to-end: about 15 minutes, ~5 of which were the agent writing the
file.

## Plugins

Org-specific patterns ship through a plugin. The matcher shape is the
same; it just lives in your plugin's `matchers/` directory and gets
registered via the plugin's `matchers: []` array. See
[plugins.md](plugins.md).

For copy-paste starting points, see
[`samples/webapp/matchers/`](../samples/webapp/matchers/) — two small
matchers (a `normal`-tier debug-flag detector and a `noisy`-tier route
glob) wired into the inline plugin in
[`samples/webapp/deepsec.config.ts`](../samples/webapp/deepsec.config.ts).
